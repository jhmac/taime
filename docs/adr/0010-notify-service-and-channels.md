# Single `notify()` service with closed type registry, four channels, generic inbox

Every notification path in the codebase goes through one service: `notify(userId, typeKey, payload)`. Feature code never calls push, email, or WebSocket broadcasts directly — that single chokepoint is what makes preferences, delivery logs, and channel fanout consistent. **NotificationType** is a closed code-defined registry (each type has a stable key, label, default channel set, and category); **NotificationChannel** is a closed enum of `in_app`, `email`, `web_push`, `native_push` — SMS is explicitly out of scope. Preferences are per-(User, Type, Channel) with "no row → use the type's default" fallback. **InboxItem** is the generic persistence for the `in_app` channel — it replaces the per-feature `commuteAlerts` and `overtimeAlerts` tables so the bell UI reads from one source. **ScheduledReport** is a separate concept from Notification (even though both use the email channel) because its cadence, recipients, and payload come from a different config surface.
