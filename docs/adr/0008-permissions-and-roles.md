# Permissions are code-defined; Roles are runtime-defined; override beats role; permissions are User-global

The **Permission** registry is closed and lives in seed code — every permission has a code site that checks it, so adding one without a code change would silently no-op. **Roles** are runtime-extensible by `owner` / `admin` (custom roles can be created and assigned in the UI), but the system roles `owner | admin | manager | employee` are always present and seeded. **PermissionOverrides** in `userPermissionOverrides` always win over Role defaults — resolution is `override row → rolePermissions for the User's Role → deny`. Permissions are not scoped per-Store or per-Location; scope is enforced at the data layer by joining through **StoreMembership** and **LocationAssignment**. This keeps the permission model flat and predictable, and pushes multi-tenant isolation into queries where it can be audited consistently.
