Create all API endpoints for the GTD Workflow Engine.

Create a new route file: `server/routes/gtd.ts`

INBOX ENDPOINTS:

1. POST /api/gtd/inbox — Quick capture to inbox
   - Auth: Any authenticated employee
   - Zod body: { raw_input: string (min 1, max 2000), source?: string }
   - Creates the inbox item, triggers async AI clarification
   - Returns the item immediately (status: 'unprocessed')
   - This must be FAST — it's the zero-friction capture point

2. GET /api/gtd/inbox — List inbox items
   - Auth: Employee sees their own captures, Manager/Owner sees all for store
   - Query params: status? ('unprocessed', 'clarified', 'processed'), limit, offset
   - Default: show 'unprocessed' and 'clarified' items sorted by created_at desc
   - Include ai_clarification data for clarified items

3. POST /api/gtd/inbox/:id/process — Process an inbox item (accept or override AI suggestion)
   - Auth: The employee who captured it, or Manager/Owner
   - Zod body: { 
       destination: 'next_action' | 'project' | 'waiting_for' | 'someday_maybe' | 'reference' | 'trash' | 'issue',
       title?: string (override AI suggestion),
       description?: string,
       // Fields specific to destination:
       context?: string, energy_level?: string, time_estimate_minutes?: number, priority?: string, 
       due_date?: string, assigned_to?: string, project_id?: string,
       waiting_on?: string, waiting_on_employee_id?: string, follow_up_date?: string,
       category?: string, tags?: string[]
     }
   - Creates the appropriate record in the destination table
   - Updates the inbox item: status = 'processed', processed_into_type, processed_into_id
   - If destination is 'issue': create an issue via the existing Issue Tracker API
   - If destination is 'trash': just mark as processed, don't create anything
   - Wrap in a transaction

4. DELETE /api/gtd/inbox/:id — Delete an inbox item
   - Auth: The capturer or Manager/Owner
   - Hard delete if unprocessed, soft delete (status: 'deleted') if already clarified

NEXT ACTIONS ENDPOINTS:

5. GET /api/gtd/actions — List next actions
   - Auth: Employee sees actions assigned to them, Manager/Owner sees all
   - Query params: context?, energy_level?, project_id?, priority?, status?, due_before?, assigned_to?, is_two_minute?
   - This is the CONTEXT-BASED view — the core of GTD. "Show me what I can do right now at the store with medium energy."
   - Sort: urgent first, then by due_date (soonest), then by priority
   - Include project title if linked to a project

6. POST /api/gtd/actions — Create a next action directly (bypassing inbox)
   - Auth: Any authenticated employee
   - Zod body: { title, description?, context?, energy_level?, time_estimate_minutes?, priority?, due_date?, assigned_to?, project_id?, is_two_minute? }

7. PUT /api/gtd/actions/:id — Update a next action
   - Auth: Assigned employee or Manager/Owner
   - Zod body: any updatable fields
   - If status changes to 'completed': set completed_at, log to activity_log

8. DELETE /api/gtd/actions/:id — Delete a next action
   - Auth: Creator or Manager/Owner

PROJECT ENDPOINTS:

9. GET /api/gtd/projects — List projects
   - Auth: Employee sees their own, Manager/Owner sees all
   - Query params: status?, owner_id?
   - Include: next action count (total and completed), progress percentage

10. POST /api/gtd/projects — Create a project
    - Auth: Any authenticated employee
    - Zod body: { title, description?, desired_outcome?, due_date?, owner_id? }

11. GET /api/gtd/projects/:id — Get project detail
    - Auth: Project owner or Manager/Owner
    - Returns project with all next actions, waiting-for items, and reference material linked to it

12. PUT /api/gtd/projects/:id — Update project
    - Auth: Project owner or Manager/Owner

WAITING FOR ENDPOINTS:

13. GET /api/gtd/waiting — List waiting-for items
    - Auth: Employee sees their own, Manager/Owner sees all
    - Query params: status?, follow_up_overdue? (boolean — follow_up_date < today)
    - Highlight overdue follow-ups

14. POST /api/gtd/waiting — Create waiting-for item
    - Auth: Any authenticated employee

15. PUT /api/gtd/waiting/:id — Update (mark as received, update follow-up date)
    - Auth: Owner of the item or Manager/Owner

SOMEDAY/MAYBE ENDPOINTS:

16. GET /api/gtd/someday — List someday/maybe items
    - Auth: Employee sees their own, Manager/Owner sees all
    - Query params: category?, status?

17. POST /api/gtd/someday — Create someday/maybe item
    - Auth: Any authenticated employee

18. PUT /api/gtd/someday/:id — Update or activate
    - Auth: Owner of the item or Manager/Owner
    - If activating: create a project or next action, update activated_into_type/id

REFERENCE ENDPOINTS:

19. GET /api/gtd/reference — List reference material
    - Auth: Employee sees their own, Manager/Owner sees all
    - Query params: search? (full-text on title + content + tags)

20. POST /api/gtd/reference — Create reference item
21. PUT /api/gtd/reference/:id — Update
22. DELETE /api/gtd/reference/:id — Delete

DASHBOARD ENDPOINT:

23. GET /api/gtd/dashboard — GTD overview counts
    - Auth: Any authenticated employee
    - Returns: { inbox_count, actions_today_count, actions_overdue_count, waiting_overdue_count, projects_active_count, two_minute_actions_count }
    - This powers the GTD widget on the main dashboard

TECHNICAL:
- Every endpoint: Clerk auth, RBAC, Zod validation, try/catch, structured logging
- All list endpoints: paginated with limit/offset
- All write endpoints: Drizzle transactions where multi-table
- WebSocket events: inbox_item_created, inbox_item_clarified, action_completed, project_completed
- Register routes in Express app
