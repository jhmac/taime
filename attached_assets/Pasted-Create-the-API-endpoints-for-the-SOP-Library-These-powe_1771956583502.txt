Create the API endpoints for the SOP Library. These power the SOP builder, library browser, and execution tracking.

Create a new route file: `server/routes/sops.ts`

ENDPOINTS:

1. POST /api/sops/templates — Create a new SOP template
   - Auth: Owner or Admin only
   - Zod body: { title, description?, category, estimated_duration_minutes?, role_assignments?, training_notes?, steps: [{ title, description?, step_type, is_checkpoint?, timer_duration_seconds?, decision_options?, training_detail? }] }
   - Creates the template AND all steps in a single transaction
   - Returns the full template with steps
   
2. GET /api/sops/templates — List all SOP templates for the store
   - Auth: Any authenticated employee
   - Query params: category? (filter), is_active? (default true), search? (text search on title/description)
   - Paginated: limit (default 20, max 50), offset
   - Returns templates with step count (not full steps)
   - For Associate role: only return templates matching their role_assignments

3. GET /api/sops/templates/:id — Get a single SOP template with all steps
   - Auth: Any authenticated employee
   - Returns template with full ordered steps
   - Include execution stats: total times executed, average completion time, last executed date

4. PUT /api/sops/templates/:id — Update an SOP template (creates a new version)
   - Auth: Owner or Admin only
   - This creates a NEW template row with version incremented and parent_template_id pointing to the old one
   - The old template's is_active is set to false
   - All steps are recreated for the new version
   - This is version control — nothing is ever deleted
   - Wrap in a transaction

5. DELETE /api/sops/templates/:id — Soft delete (set is_active = false)
   - Auth: Owner only

6. POST /api/sops/executions — Start an SOP execution
   - Auth: Any authenticated employee
   - Zod body: { template_id }
   - Creates an execution record and pre-populates sop_step_completions for all steps as 'pending'
   - Returns the execution with all step completions
   - Log to activity_log

7. PUT /api/sops/executions/:id/steps/:stepId — Complete or skip a step
   - Auth: The employee who started the execution, OR a manager
   - Zod body: { status: 'completed' | 'skipped', skip_reason?, photo_url?, notes?, time_spent_seconds? }
   - If the step is a checkpoint and requires manager sign-off, set manager_sign_off to false and notify the manager via WebSocket
   - If all steps are completed/skipped, auto-update the execution status to 'completed' and set completed_at

8. PUT /api/sops/executions/:id — Update execution status
   - Auth: The employee or their manager
   - Zod body: { status: 'completed' | 'abandoned' | 'paused', notes? }
   - Log to activity_log

9. GET /api/sops/executions — List executions
   - Auth: Employee sees their own, Manager/Owner sees all for the store
   - Query params: employee_id?, template_id?, status?, date_from?, date_to?
   - Paginated
   
10. POST /api/sops/templates/:id/sign-off/:stepCompletionId — Manager signs off on a checkpoint
    - Auth: Manager or Owner only
    - Sets manager_sign_off = true, manager_sign_off_by, manager_sign_off_at

TECHNICAL REQUIREMENTS:
- Every endpoint: Clerk auth middleware, RBAC check, Zod validation, try/catch, structured logging
- All list endpoints: pagination with limit/offset
- All write endpoints: wrap in Drizzle transaction
- Response shape: { success: true, data: {...} } or { success: false, error: { message, code } }
- Emit WebSocket events for: execution_started, step_completed, execution_completed, sign_off_requested
- Register routes in the main Express app

TEST: After building, verify you can create a template with steps, start an execution, complete steps one by one, and see the execution complete automatically.
