### Part A: Add `createFile` method to Code Engine

**File: `src/code-engine.js`**

Read this file first. Look at how `applyChange`, `backup`, and `rollback` work. Your new methods should follow the same patterns.

**Check the imports at the top.** The file probably uses `const fs = require('fs')` (sync). You'll need the promises version for async operations. If only sync `fs` is imported, add `const fsPromises = require('fs').promises;` and use `fsPromises` in the new methods. Do NOT rename the existing `fs` import.

Add these two methods to the CodeEngine class (or module, depending on how it's structured):

#### `createFile(filePath, content, projectRoot)`

This creates a brand new file. It should:

1. Resolve the full path using `path.resolve(projectRoot, filePath)`
2. Run the EXISTING `_checkSafety` method on the full path — same safety checks as editing
3. Check if the file ALREADY EXISTS — if it does, return `{ success: false, error: 'File already exists: ${filePath}. Use applyChange to modify existing files.' }`. Use `fs.existsSync` or `fsPromises.access` with a try/catch.
4. Create parent directories if they don't exist — `fsPromises.mkdir(dir, { recursive: true })`
5. Write the file content — `fsPromises.writeFile(fullPath, content, 'utf-8')`
6. If the file extension is .js, .ts, .jsx, or .tsx, run the EXISTING `verifySyntax` method on it
7. If syntax verification fails, DELETE the file you just created (don't leave broken files) and return `{ success: false, error: 'Syntax errors: ...' }`
8. Return `{ success: true, filePath: fullPath }`

#### `deleteFile(filePath)`

This deletes a file (used for rollback when a created file needs to be undone). It should:

1. Try to `fsPromises.unlink(filePath)`
2. Return `{ success: true }` or `{ success: false, error: error.message }`

This is intentionally simple — it's a cleanup method for rollback, not a general-purpose delete.

### Part B: Add "create" handling to Ralph Loop

**File: `src/ralph-loop.js`**

Read this file first. Find the main execution loop in `executeRalphLoop` (or whatever the main function is called). Look at how it handles the spec executor's response — there should be conditions for:
- `SPEC_COMPLETE`
- `result.status === 'change'`
- `result.status === 'multi-change'`
- `result.status === 'stuck'`

Add handling for two new status types. Add these BEFORE the existing `'change'` handler:

#### Handle `status: 'create'` (single file creation)

When the spec executor returns `{ status: 'create', filePath: '...', content: '...' }`:

1. Call `codeEngine.createFile(result.filePath, result.content, projectRoot)`
2. If creation fails → set iteration status to `'create-failed'`, increment stuck counter
3. If creation succeeds → run test command if the spec has one (same pattern as existing change tests)
4. If tests fail after creation → delete the created file (rollback), set status to `'create-failed'`
5. If runtime validation is configured → run it (same pattern as existing)
6. If runtime fails → delete the created file, set status to `'create-failed'`
7. If everything passes → set iteration status to `'created'`, reset stuck counter
8. Record `filesCreated: [result.filePath]` in the iteration history

#### Handle `status: 'multi-create'` (multiple file creation)

When the spec executor returns `{ status: 'multi-create', files: [{ filePath, content }, ...] }`:

1. Create files ONE AT A TIME in order
2. Track which files were successfully created in a `createdFiles` array
3. If ANY file fails to create → delete ALL previously created files in this batch (rollback) → return failure
4. After all files created → run test command if specified
5. If tests fail → delete ALL created files → return failure
6. If everything passes → set iteration status to `'created'`

**The key principle is atomic operations:** either ALL files in a multi-create succeed, or NONE of them persist. Same as the existing `_applyMultiFileChanges` pattern.
