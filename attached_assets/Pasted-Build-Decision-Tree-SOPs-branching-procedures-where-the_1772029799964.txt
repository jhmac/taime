Build Decision Tree SOPs — branching procedures where the next step depends on a decision or condition.

BACKEND:

1. The SOP schema already supports decision steps (step_type: 'decision' with decision_options jsonb from Phase 1). This prompt makes the builder and execution UI fully support branching.

2. Enhance the decision_options schema:
   
   Current: { "options": [{ "label": "Yes", "next_step_order": 5 }, { "label": "No", "next_step_order": 6 }] }
   
   Enhanced:
   {
     "question": "Is the customer returning within 30 days of purchase?",
     "options": [
       { "label": "Yes, within 30 days", "next_step_order": 5, "color": "green" },
       { "label": "No, past 30 days", "next_step_order": 8, "color": "yellow" },
       { "label": "No receipt", "next_step_order": 11, "color": "red" }
     ]
   }
   
   - Support 2-5 options per decision
   - Each option can jump to any step in the SOP (not just the next one)
   - Options can have color hints for the UI (green = good path, red = exception path)

3. API changes:
   - The existing SOP template creation/update endpoints already accept decision_options
   - Add validation: ensure all next_step_order values reference actual steps in the template
   - Add a new endpoint: GET /api/sops/templates/:id/flow — Returns the SOP as a graph (steps as nodes, decisions as branching edges) for visualization

FRONTEND:

4. SOP Builder — Decision Step Editor:
   
   When a step has step_type = 'decision':
   - Show the decision question field: "What question are you deciding?"
   - Options list:
     - Each option has: label (text), target step (dropdown of all steps), color (optional)
     - "Add Option" button (max 5)
     - Remove option button per option
   - Visual indicator: show arrows from this decision to target steps
   - Validation: warn if any option targets a step that doesn't exist

5. SOP Builder — Flow Visualization:
   
   Add a "View Flow" toggle in the SOP builder:
   - Switches from the linear step list to a visual flowchart
   - Use a simple tree/graph layout:
     - Action/verification/photo/timer steps: rectangular nodes in a vertical chain
     - Decision steps: diamond-shaped nodes with branching paths to target steps
     - Lines connecting steps show the flow
   - This helps the owner visualize the branching logic
   - Use a library: react-flow, dagre, or simple SVG drawing
   - Can be read-only in MVP (editing happens in the list view)

6. SOP Execution UI — Decision Steps:
   
   When the execution reaches a decision step:
   - Display the decision question prominently
   - Show option buttons (large, tappable, color-coded if colors are set)
   - Employee taps their choice
   - The execution JUMPS to the target step (skipping intervening steps if needed)
   - Breadcrumb trail shows which path was taken
   - Steps that were skipped due to branching are NOT marked as "skipped" — they simply don't apply to this path
   
   SOP completion logic:
   - A branching SOP is "complete" when the employee reaches the END of their branch
   - Track which path was taken in the sop_step_completions (add a field: branch_path jsonb — array of decision choices made)

7. Example Decision Tree SOPs to seed:
   
   "Return Processing" SOP:
   - Step 1: Greet customer, ask about the return
   - Step 2: DECISION: "Does the customer have a receipt?"
     - Yes → Step 3
     - No → Step 6
   - Step 3: DECISION: "Is it within 30 days?"
     - Yes → Step 4 (Full refund)
     - No → Step 5 (Store credit only)
   - Step 4: Process full refund, end
   - Step 5: Process store credit, end
   - Step 6: DECISION: "Can you find the transaction in Shopify?"
     - Yes → Step 3 (back to 30-day check)
     - No → Step 7
   - Step 7: Manager approval required, end

TECHNICAL:
- Decision branching must be handled in the execution logic — the "next step" after completing a decision is determined by the chosen option, not by step_order
- The SOP execution API needs to handle non-linear step progression
- Track the actual path taken through the SOP for analytics (which branch is most common?)
- Flow visualization: keep it simple in MVP — fancy interactive graph editors are Phase 4 territory
